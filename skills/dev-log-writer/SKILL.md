# 开发日志撰写规范

## 第零步：确定今天的日期（必须执行）

**AI 内置的日期知识不可靠，必须通过终端命令获取用户系统的真实时间：**

```bash
date '+%Y-%m-%d'
```

使用命令返回的日期作为今天日期，用户明确指定了日期时以用户指定为准。

## 存储位置

- 目录：`docs/`
- 文件命名：`YYYY-MM-DD.md`（如 `docs/2026-03-17.md`）
- 每天一个文件；如果当天文件已存在，追加内容而非覆盖

## 文件结构

```markdown
# YYYY-MM-DD 开发日志

## 今日实现

### N. 功能名称

**背景/问题：** 为什么需要做这件事。

**涉及文件：**
- `路径/文件.ts` — 改了什么、为什么改

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案 A | ... | ... |
| **方案 B（选用）** | ... | ... |

选型理由：一句话说明为什么选这个方案。

**实现要点：** 关键实现细节、设计决策、踩坑记录。

---

## 关键收获

本次开发中提炼的经验、设计洞见或原则，编号列出。
```

## 撰写规则

1. **技术选型必须写**：凡是做了非显而易见的设计决策，必须列出备选方案和最终选择的理由
2. **涉及文件要标注变更说明**：每个文件后用破折号说明"改了什么"，不只列文件名
3. **关键收获放在文件最底部**：提炼本次开发中能跨越具体功能复用的经验
4. 用具体的技术语言描述，避免"优化了性能"这类模糊表达
5. 非显而易见的实现模式可以附代码块说明
6. 每个功能小节应自成一体，能独立阅读理解

## 第一步：撰写开发日志

按照上述文件结构撰写 `docs/YYYY-MM-DD.md`，内容要求见撰写规则。

## 第二步：更新 README.md

**开发日志写完后，必须更新项目根目录的 `README.md`，确保 README 与最新功能同步。**

需要更新的内容：
- **后续计划**：新完成的功能勾选（`- [ ]` 改为 `- [x]`）
- **工具表格**：有新增工具时，追加到对应 Agent 的工具表格中
- **技术栈**：有新引入的技术时，加入技术栈列表
- **快速上手**：有新配置项时追加配置步骤说明

## 第三步：提交并推送

**内容确认无误后，执行以下命令提交并推送到远程仓库：**

```bash
# 1. 暂存所有改动（不包括未跟踪文件）
git add -A

# 2. 提交（commit message 参考本次开发内容，简洁清晰）
git commit -m "feat: 干了什么、干了什么"

# 3. 推送到当前分支
git push
```

commit message 格式：使用 `feat:` / `fix:` / `docs:` / `refactor:` 前缀，简洁说明本次主要内容。
