## 今日实现

### 1. 全局工具注册架构重构（OpenClaw 风格）

**背景/问题：** 工具按 Agent 类别拆分注册（`createAllCodingTools` / `ALL_DAILY_TOOLS`），路由出错时整套工具不可用。更深层问题：工具按 Agent 分类本身就是错误的抽象——工具是能力，不是 Agent 的专属财产。

**涉及文件：**
- `src/tools/registry.ts` — 新建；全局工具注册表，包含所有 14 个工具的 `def` + `impl` + `readonly`；`CODING_AGENT_DENY` / `DAILY_AGENT_DENY` 两个黑名单常量；`registerWithDenyList()` 辅助函数
- `src/agents/coding/index.ts` — 移除 `createAllCodingTools`，改为 `createGlobalToolRegistry(workDir) + registerWithDenyList(CODING_AGENT_DENY)`
- `src/agents/daily/index.ts` — 移除 `ALL_DAILY_TOOLS`，改为 `createGlobalToolRegistry + registerWithDenyList(DAILY_AGENT_DENY)`；新增 `workDir` 参数；清理 system prompt 中的"请转交 coding_agent"硬编码路由限制
- `src/cli/index.ts` — 移除 `ALL_CODING_TOOLS` / `ALL_DAILY_TOOLS` 导入；`/tools` 命令改为 `createGlobalToolRegistry(CWD)` 动态枚举；`createDailyAgent` 调用补传 `CWD`

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 白名单（allow list）：Agent 只能用声明的工具 | 权限精确 | 路由出错 = 工具缺失，Agent 退化为普通对话 |
| **黑名单（deny list）：全局工具池 - 禁止项（选用）** | 路由出错最多多几个无用工具，核心能力不受影响 | 每个 Agent 需明确列出不该有的工具 |

选型理由：Deny list 路由鲁棒性更强；"工具注册是全局的，权限控制是局部的"——参照 OpenClaw 架构原则。

**实现要点：**
- `ToolEntry.impl` 需声明为 `ToolFunction`（`(args: any) => string | Promise<string>`），而非 `Promise<string>`，否则同步工具实现无法赋值
- Daily Agent 原本靠系统 prompt 中的"请告知用户转交 coding_agent"来限制跨域行为，全局注册后改为 deny list 直接控制，系统 prompt 中的路由限制文案一并删除，更干净
- 工具分配结果：Coding Agent（9 个）= 全集 - Daily 专属工具；Daily Agent（9 个）= 全集 - Coding 专属工具

- `src/agents/daily/tools/`（整目录）— `git rm` 删除，共 6 个文件（含 index.ts）

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 保留 agents 子目录，registry.ts 跨目录引用 | 改动最小 | 认知割裂：全局注册 + 文件分散 |

## 关键收获

1. **Deny list 比 Allow list 更鲁棒**：路由出错时，deny list 最多让 Agent 多几个无用工具，allow list 则可能让 Agent 丢失关键能力。对于 Agent 系统，宁可工具略多，不要工具缺失。
2. **工具注册是全局的，权限控制是局部的**：这是 OpenClaw 风格的核心原则。工具定义不应绑定到任何 Agent，Agent 只通过黑名单声明"我不需要什么"。
3. **架构语义一致性**：全局注册表建立后，工具文件也必须对齐到同一目录。"注册在 A，文件在 B"是认知负担，应在重构完成时一并消除。
4. **批量路径修复 + 即时编译验证**：迁移多文件时，`sed -i ''` 批量处理 + `tsc --noEmit` 是标准流程，缺一不可。
5. **AI 日期不可信**：Skill 中必须显式执行 `date` 命令获取真实日期，本次遗漏导致开发日志被写入错误日期，发现后立即修正。
