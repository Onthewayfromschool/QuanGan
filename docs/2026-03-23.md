# 2026-03-23 开发日志

## 今日实现

### 1. Coding Agent 路径安全守卫

**背景/问题：** Coding Agent 执行 `rm`/`mv`/`cp`/`rmdir` 等危险命令时没有任何约束，一旦 Agent 生成的命令路径超出项目目录，可能造成不可逆的文件损失。

**涉及文件：**
- `src/agents/coding/tools/execute-command.ts` — 静态 `implementation` 改为工厂函数 `createImplementation(workDir, confirmFn?)`；新增 `extractPaths()` 提取命令中的路径参数，`hasOutsidePath()` 用 `path.resolve` 规范化后比对 `workDir` 前缀，越界时调用 `confirmFn` 暂停等待用户 y/N 输入
- `src/agents/coding/tools/index.ts` — `ALL_CODING_TOOLS` 改为工厂函数 `createAllCodingTools(workDir, confirmFn?)`，内部调用 `createImplementation`
- `src/agents/coding/index.ts` — `callbacks` 新增 `confirm?: (msg: string) => Promise<boolean>` 字段，传入工具工厂
- `src/cli/index.ts` — 新增 `makeConfirmFn()`，通过闭包捕获模块级 `rlInstance`，在终端输出黄色警告并等待 stdin 一行输入

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 直接拒绝越界命令 | 最安全，零风险 | 合法的跨目录操作也会被拦截 |
| **注入 confirmFn 询问用户（选用）** | 灵活，合法越界可放行 | 需要 readline 交互设计 |

选型理由：部分场景确实需要操作项目外路径，直接拒绝过于强硬；y/N 询问兼顾安全与灵活性。

**实现要点：** `confirmFn` 通过闭包捕获 `rlInstance`，时序安全（main() 设置好 rl 之后工具才会被调用）。路径检测对命令第一个词判断是否属于危险操作，再提取非 flag 参数做路径规范化比对。

### 2. verify_code 编译验证工具

**背景/问题：** Agent 写完代码后只能靠语义判断是否正确，无法自动检验类型错误，需要人工跑 `tsc` 确认。

**涉及文件：**
- `src/agents/coding/tools/verify-code.ts` — 新建；运行 `npx tsc --noEmit`，统计 `error TS` 行数，编译通过返回成功提示，失败返回完整错误信息
- `src/agents/coding/tools/index.ts` — 注册 `verify_code` 工具（readonly: false）
- `src/agents/coding/index.ts` — 系统提示新增一句：修改代码后可调用 `verify_code` 自我检验

**实现要点：** `tsc` 报错时以非 0 退出码退出，错误输出在 `e.stdout`（不在 stderr），需从 `e.stdout` 而非 `e.stderr` 读取；`execSync` 超时设 60s，因为大项目 tsc 可能较慢。

### 3. execute_command 退出码 1 误判修复

**背景/问题：** `grep` 无匹配时以退出码 1 退出，`execSync` 把非零退出码一律当命令失败抛异常，导致 `sleep 3 && ps aux | grep "npm run dev" | grep -v grep` 这类正常查询命令误报失败。

**涉及文件：**
- `src/agents/coding/tools/execute-command.ts` — catch 块新增 `e.status === 1` 特判：stdout 有内容则返回 stdout，无内容则返回 `(命令已执行，无匹配结果)`；退出码 2+ 才视为真正失败

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 命令末尾加 `\|\| true` | 不改代码 | 每次需用户手动处理 |
| **源码特判退出码 1（选用）** | 一次修复，永久生效 | 需区分 grep 无匹配与真正失败 |

选型理由：统一在工具层处理，对 Agent 和用户完全透明，无需每次修改命令。

**实现要点：** `e.status === 1` 且 `e.stdout` 有内容 → 返回 stdout（grep 有输出但无匹配）；`e.status === 1` 且无 stdout → 返回友好提示；`e.status >= 2` 或有 stderr → 返回错误。

## 关键收获

1. **工厂函数 + 依赖注入**：工具的危险行为（路径越界确认）不应硬编码在工具内部，而应通过工厂函数注入外部回调，调用方决定如何处理——这是工具层与 UI 层解耦的标准模式
2. **退出码语义不等于执行成功**：`grep`/`diff` 等工具用退出码 1 表示"无结果"而非"失败"，`execSync` 的异常处理必须区分退出码 1（无结果）和 2+（真正失败）
3. **tsc 错误输出在 stdout 不在 stderr**：TypeScript 编译错误写入 stdout，用 `execSync` 时需从 `e.stdout` 读取，而非 `e.stderr`
