- `src/tools/verify-code.ts` — 同上

- `src/agents/coding/tools/`（整目录删除）— `git rm` 共 10 个文件（含 index.ts）
- `src/agents/daily/tools/`（整目录删除）— `git rm` 共 6 个文件（含 index.ts）

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 保留 agents 子目录，registry.ts 跨目录引用 | 改动最小 | 认知割裂：全局注册 + 文件分散 |
| **工具全部迁移到 src/tools/（选用）** | 文件位置与架构语义完全一致 | 需修复文件内 types.js 相对路径 |

选型理由：全局工具注册表已是架构核心，工具文件理应与之同目录，消除"代码在 agents/，但入口在 tools/"的双重定位。

**实现要点：**
- 迁移前文件内的 `from '../../../tools/types.js'`（从 `agents/X/tools/` 三级上跳）迁移后需改为 `./types.js`
- 用 `sed -i ''` 批量替换 14 个文件，一条命令搞定，避免逐文件手改
- 删除旧目录使用 `git rm` 而非 `rm`，确保删除操作进入 git staging，版本历史可追溯
- 迁移后立即 `npx tsc --noEmit` 验证，零错误确认

### 2. 全局 daily-record Skill 同步更新

**背景/问题：** 全局 `~/.qoder/skills/daily-record/SKILL.md` 版本落后于项目版本，缺少 3 个关键步骤：第零步（获取真实日期）、第三步（更新 README）、第四步（git 推送）。导致在其他项目使用该 skill 时，AI 依赖内置日期知识（不可靠），且不会自动收尾推送。

**涉及文件：**
- `~/.qoder/skills/daily-record/SKILL.md` — 新增第零步、第三步、第四步；删除旧版"日期默认使用今天"模糊说明

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 手动维护两份文件 | 无 | 容易遗漏，版本碎片化 |
| **以项目版本为准覆盖全局（选用）** | 项目是主要开发场景，改动在项目测试后同步全局 | 多项目时需注意差异 |

选型理由：skill 的主要测试场景在当前项目，项目版本经过实际使用验证后再同步到全局，比双向同步更简单。

**实现要点：**
- 新增「第零步」的核心价值：`date '+%Y-%m-%d'` 强制使用系统真实时间，解决 AI 日期幻觉问题（今天的记录就曾被写入错误日期 2026-02-16）
- 第三步 README 更新和第四步 git 推送是"收尾闭环"，确保每次 daily-record 都能形成完整可追溯的提交

## 关键收获

1. **架构语义一致性**：工具注册全局化后，工具文件也必须对齐到同一目录。"注册在 A，文件在 B"是认知负担，应在重构完成时一并消除。
2. **批量路径修复 + 即时编译验证**：迁移多文件时，`sed -i ''` 批量处理 + `tsc --noEmit` 是标准流程，缺一不可。
3. **git rm 而非 rm**：删除已 track 的源文件必须用 `git rm`，否则 git status 会显示文件消失但未暂存，容易遗漏。
4. **AI 日期不可信**：skill 中必须显式执行 `date` 命令获取真实日期，本次遗漏此步骤导致开发日志和发言归档被写入了错误日期（2026-02-16），发现后立即修正。
