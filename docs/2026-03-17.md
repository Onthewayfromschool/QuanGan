### 2. 上下文压缩（Rolling Summary）

**问题：** 随着对话轮数增多，`messages` 数组持续膨胀，发送给 LLM 的 token 越来越多，最终撞上模型上下文上限（即便是 qwen3.5-plus 的 1,000,000 token 也是有限的），且费用线性增长。

#### 2.1 压缩策略技术选型

在实现前调研了几种常见方案：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 滑动窗口（Sliding Window） | 只保留最近 N 条消息 | 实现最简单 | 旧对话上下文彻底丢失，模型看不到任何历史 |
| **滚动摘要（Rolling Summary，选用）** | 旧消息调用 LLM 生成摘要后注入，保留最近 N 条 | 历史信息有损但保留，LLM 仍有完整感知 | 摘要准确性依赖压缩 prompt 质量 |
| 层级摘要（Hierarchical） | 多轮摘要叠加，保留层级结构 | 长对话效果更好 | 实现复杂，需维护多层摘要树 |
| Token 预算（Token Budget） | 精算每条消息 token 数，按预算动态裁剪 | 非常精准 | 依赖 tokenizer 库，工程复杂 |

选用 **滚动摘要**，理由：
- 对于 Coding Agent 场景，旧消息里的"已读过哪些文件、做过哪些修改"是最关键的信息，摘要能有效保留
- 实现简单直接，无需额外依赖
- "激进策略"进一步降低 token 占用：LLM 上下文只使用最近一次摘要 + 之后的消息，不需要链式摘要叠加


**触发时机：** 每次 API 响应后读取 `data.usage.total_tokens`，超过阈值（默认 16,000）时自动触发。选择"响应后"而非"请求前"的原因：百炼返回的是精确值，无需估算，且此时刚好有一个完整的请求-响应单元可作为压缩边界。

**示例**
旧: [system] [user1] [ai1] [tool] [user2] [ai2] ... [user10] [ai10]
     ↓ 触发压缩（保留最近3轮）
新: [system] [summary: 前7轮的核心信息] [user8] [ai8] [user9] [ai9] [user10] [ai10]

#### 2.2 消息格式设计选型

这是今天花时间最多的一个设计决策，从双数组方案迭代到最终的单数组标记方案。

**核心问题：** 压缩后 LLM 上下文变短了，但用户打开 `/history` 应该还能看到完整的对话记录，两者诉求矛盾。

**方案一：双数组（最初实现）**

```
displayMessages  ← 完整历史，只增不减，用于 /history 展示
messages         ← LLM 上下文，压缩时被裁剪
```

- **优点：** 职责清晰，概念上容易理解
- **缺点：** 每次 `push` 要同步维护两个数组，容易漏写；存档文件要存两份（`display` + `context`）；加载时要分别恢复两个数组；同步问题一旦出现很难排查

**方案二：单数组 + 标记位（最终选用）**

只维护一个 `messages` 数组，通过元数据字段区分消息状态：

```
_archived: true  ← 已被压缩归档，保留在数组中供 /history，但不发给 LLM
_summary: true   ← 压缩摘要节点，是 LLM 重建上下文的锚点
```

压缩发生时：
1. 旧消息原地打上 `_archived: true`，不删除（保留历史）
2. 在"保留消息"之前插入摘要节点（`_summary: true`）

LLM 上下文派生（`getLLMMessages()`）：
- 找到数组中最后一个 `_summary` 节点
- 取 `system_prompt` + 该摘要节点之后的所有消息
- 剥离 `_archived` / `_summary` 字段后发给 API（LLM 看到的是干净的消息列表）

**激进策略的设计考量：** 每次压缩后只保留最近一次摘要，不需要链式累加。多轮压缩后数组里会有多个摘要节点，但 `getLLMMessages()` 总是只用最后一个——这样每次发给 LLM 的 token 量始终稳定在"一个摘要 + 最近 6 条消息"的量级，无论对话多长都不会膨胀。

**方案对比：**

| 维度 | 双数组 | 单数组+标记（选用） |
|------|--------|---------------------|
| 同步风险 | 高（两处 push） | 无 |
| 存档复杂度 | 两份数据（`display` + `context`） | 一份（全量 messages） |
| 加载逻辑 | 需分别恢复两个数组 | 一次加载，标记自动生效 |
| 可读性 | 调试时要对比两个数组 | 单数组完整呈现历史 |
| 格式扩展性 | 差（加新状态需新增数组） | 好（加字段即可） |

**涉及文件：**
- `src/llm/types.ts` — `ChatMessage` 接口新增 `_archived?: boolean` 和 `_summary?: boolean` 可选字段
- `src/agent/agent.ts` — 新增 `getLLMMessages()` 派生方法；重写 `compressContext()` 改为原地标记；`run()` 改用 `getLLMMessages()` 发送请求
- `src/cli/session-store.ts` — `saveSession` 过滤掉非摘要的 system 消息，保留 `_summary` 和普通角色消息；`loadSession` 直接加载单数组

### 3. 压缩体验优化

**问题：** 压缩需要额外调用一次 LLM，耗时可能数秒，期间终端没有反馈，用户以为程序卡死。

**方案：** 在 `AgentConfig` 中增加两个回调：

| 回调 | 触发时机 | 用途 |
|------|----------|------|
| `onCompressStart` | 压缩开始前 | TUI 层输出黄色等待提示 |
| `onCompress(before, after)` | 压缩完成后 | 覆盖等待行，输出完成摘要 |

效果：
```
  ⏳ 上下文过长，正在压缩历史对话...     ← 黄色，压缩期间显示
[System] ♻️  上下文已自动压缩（28 → 10 条消息）  ← 完成后覆盖
```

回调设计在 `AgentConfig` 而非内部 logger，目的是职责分离：Agent 核心不感知 TUI，TUI 层决定如何展示。

## 关键收获

1. **单数组 + 标记位是 LLM 对话历史管理的优雅解法**：将"展示视图"和"LLM 视图"的区别放在派生逻辑里，而非在数据结构层维护两份副本，消除了同步问题的根源。

2. **"激进策略"（只用最近一次摘要）比链式摘要更实用**：每次发给 LLM 的 token 量有上界，不会随对话长度线性增长，且实现极简。

3. **压缩触发用 API 返回的 `total_tokens` 而非估算**：百炼接口精确返回 token 数，比字符数 ÷ 4 的粗略估算更可靠，触发时机更准确。

4. **回调 vs 日志**：将压缩事件通过回调暴露给上层，而非在 Agent 内直接 `console.log`，使得 Agent 核心与 TUI 展示彻底解耦，也方便未来接入其他 UI 形态。
