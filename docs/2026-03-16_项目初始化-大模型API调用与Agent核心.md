### 2. Agent 工具调用（Function Calling）

**涉及文件：**
- `src/tools/types.ts` — 工具相关类型（ToolDefinition、ToolCall、ToolResult）
- `src/agent/agent.ts` — Agent 核心，实现 ReAct 循环：
  1. 用户消息 push 进 messages
  2. 发给大模型，携带 tools 定义
  3. 若模型返回 tool_calls → 执行工具 → 把结果 push 进 messages → 再次调用模型
  4. 模型不再需要工具时返回最终答案

**关键知识点：**
- messages 数组里每条消息通过 `role` 区分身份：
  - `system` — 系统提示（构造时写入）
  - `user` — 用户输入
  - `assistant` — 模型回复（API 返回时自带 role 字段）
  - `tool` — 工具执行结果
- 每次调用 API 都把**全量 messages 历史**一起发送，模型靠此维持上下文
- 通过 `onToolCall` / `onToolResult` 回调将执行过程暴露给上层（TUI 展示用）

### 4. Coding 工具集（模块化）

每个工具独立一个文件，统一导出 `definition`（告诉模型）和 `implementation`（实际执行）：

| 文件 | 工具名 | 功能 |
|------|--------|------|
| `src/cli/tools/read-file.ts` | `read_file` | 读取文件，支持指定行范围 |
| `src/cli/tools/write-file.ts` | `write_file` | 写入/创建文件，自动建目录 |
| `src/cli/tools/list-directory.ts` | `list_directory` | 列出目录内容 |
| `src/cli/tools/execute-command.ts` | `execute_command` | 执行 shell 命令（带安全黑名单，支持后台模式） |
| `src/cli/tools/search-code.ts` | `search_code` | 递归搜索关键词，支持正则 |
| `src/cli/tools/index.ts` | — | 汇总导出 ALL_CODING_TOOLS |

**扩展方式：** 新建工具文件 → 在 `index.ts` 追加到数组即可

工具内部分两种模式，解决 `execSync` 不适合长驻进程的问题：

| 模式 | 参数 | 底层实现 | 适用场景 |
|------|------|----------|----------|
| 普通模式 | `background: false`（默认） | `execSync` 同步阻塞 | `ls`、`git status` 等短命令 |
| 后台模式 | `background: true` | `spawn + detached` 完全解绑 | `npm run dev`、`python app.py` 等长驻进程 |

根本原因：长驻进程永不退出，`execSync` 等到 15s 超时就抛错，Agent 误判为失败。改用 `spawn detached` 后立即返回 PID，子进程独立运行，Agent 退出也不影响它。

### 5. 跨目录启动

**问题：** 默认只能在项目目录下 `npm run cli`，换目录后 `.env` 读不到

**解决方案：**
- `bin/coding-agent.js` 作为全局入口，用 `__dirname` 定位项目位置
- `src/cli/index.ts` 的 dotenv 改为 `path.resolve(__dirname, '../../.env')`，固定从项目自身加载 Key，与 `process.cwd()` 解耦
- `package.json` 加 `bin` 字段声明可执行命令
- 配置 shell alias：`alias coding-agent="node /path/to/bin/coding-agent.js"`

**使用方式：**
```bash
cd ~/任意项目目录
coding-agent   # Agent 在该目录下工作，Key 从 baseAIlearn 读取
```

### 6. 会话历史持久化

**问题：** Agent 进程退出后，对话记录全部丢失，每次启动都是全新开始

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| JSON 文件（选用） | 零依赖、透明可读、10 分钟实现 | 不支持关键词检索 |
| SQLite | 结构化查询、可搜索 | 需引入 `better-sqlite3` 依赖 |
| JSONL | 追加写性能好 | 随机读取麻烦 |

选用 **JSON 文件**方案，适合当前学习定位：存储文件可直接打开观察，结构清晰。

**实现方案：**
- 新增 `src/cli/session-store.ts` 负责所有 IO 操作
- 存储路径：`<agent项目>/.sessions/<项目名>-<路径hash前8位>.json`
  - 例：`.sessions/vibeCoding-5b7c65fc.json`
  - 每个工作目录独立一份文件，互不干扰
- 只存 `user / assistant / tool` 消息，`system` prompt 每次启动时重新注入
- 新增 `Agent.loadMessages()` 方法，将历史消息注入到 system prompt 之后

**生命周期：**
- **启动时**：检测当前 cwd 的 session 文件，有则加载并展示历史 + 提示轮数
- **每次回复后**：自动写盘（防止进程崩溃丢失）
- **`/clear` 命令**：同时清空内存 + 删除 session 文件

**`.gitignore` 补充：** `.sessions/` 目录不纳入版本控制

### 7. Plan 模式

**背景：** 默认 Agent 是"边想边干"——收到任务直接调用工具执行。Plan 模式让 Agent 先分析、输出规划，确认后再执行，避免误操作。

**技术选型：**

| 方案 | 原理 | 复杂度 | 效果 |
|------|------|--------|------|
| 提示词注入 | system prompt 加"先规划再执行"，不限制工具 | 最简单 | 不可控，模型可能直接执行 |
| **模式标志位（选用）** | `/plan` 进入规划模式，隐藏写操作工具，前置规划指令 | 中等 | 清晰可控，读写明确分离 |
| 双阶段 Agent | 每次任务先跑"只思考" Agent，再跑"执行" Agent | 较复杂 | 最接近 Claude Code，工程量大 |

选用**模式标志位**方案：通过工具分级（`readonly` 标记）限制 Agent 能做什么，再通过提示词注入引导输出规划格式，两者结合达到可控效果。

**交互方式：**
```
> /plan          # 进入 Plan 模式，提示符变为黄色 [PLAN] >
[PLAN] > <任务>  # Agent 读代码、分析，输出结构化计划，不会写文件
[PLAN] > /exec   # 切回执行模式，提示符恢复绿色 >
> <同一任务>     # 这次才真正执行
```

**工具分级设计（核心）：**

Plan 模式不是禁用所有工具，而是只暴露只读工具给 LLM：

| 工具 | readonly | Plan 模式 |
|------|----------|-----------|
| `read_file` | ✅ | 可用 |
| `list_directory` | ✅ | 可用 |
| `search_code` | ✅ | 可用 |
| `write_file` | ❌ | 隐藏 |
| `execute_command` | ❌ | 隐藏 |

LLM "看不到"的工具自然不会调用，无需在实现层拦截。

**提示词注入：**

仅有工具限制不够，LLM 不知道要输出规划格式。在 Plan 模式下，发给 LLM 的消息会前置注入指令：

```
[当前处于规划模式，你只能使用只读工具分析代码，禁止修改任何文件]
请分析完成后，输出格式如下：
📋 执行计划
Step 1: ...
Step 2: ...
用户任务：<原始输入>
```

注入发生在**用户侧**（不修改 system prompt），`/history` 展示的仍是用户原始输入，不会污染对话记录。

**涉及文件：**
- `src/cli/tools/index.ts`：每个工具加 `readonly: boolean` 标记
- `src/agent/agent.ts`：`registerTool()` 增加 `readonly` 参数；`getToolDefinitions(planOnly)` 按模式过滤；`run()` 增加 `planOnly` 参数
- `src/cli/display.ts`：新增 `printModeSwitch()` 展示模式切换提示；`/help` 补充 `/plan` 和 `/exec` 命令
- `src/cli/index.ts`：`isPlanMode` 标志位；`/plan` / `/exec` 命令路由；`messageToSend` 注入规划指令；动态切换 readline prompt 颜色
