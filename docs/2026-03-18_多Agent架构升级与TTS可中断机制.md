### 2. QuanGan 多 Agent 架构升级（工具型）

**背景/问题：** 项目原本只有单一 Coding Agent，定位限于代码任务。为了扩展成通用个人助手（开发任务 + 日常任务），需要引入多 Agent 架构。

**技术选型：**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 路由型（Router） | 主 Agent 做意图分类，转发给子 Agent | 结构清晰，边界明确 | 跨 Agent 复合任务处理麻烦 |
| **工具型（Sub-agents as Tools，选用）** | 子 Agent 注册为主 Agent 的 Function Call 工具 | 最灵活，主 Agent 自主决策，可组合多个子 Agent | token 消耗相对高，主 Agent 调度复杂度增加 |
| 进程型（Subprocess） | 子 Agent 作为独立进程，IPC 通信 | 强隔离 | 进程通信协议复杂，调试困难 |
| 事件驱动型 | 消息队列解耦 | 适合大规模分布式 | 杀鸡用牛刀，学习成本高 |

选型理由：工具型直接复用现有 Function Calling 机制，与 Agent 核心代码无缝融合，子 Agent 可无状态调用，实现成本最低且最灵活。

**涉及文件：**
- `src/agents/coding/index.ts` — 新建，`createCodingAgent()` 工厂函数，接收 workDir 和 TUI 回调
- `src/agents/coding/tools/` — 新建目录，从 `src/cli/tools/` 整体迁移 6 个工具文件，更新 import 路径（`../../tools/types` → `../../../tools/types`）
- `src/agents/daily/index.ts` — 新建，`createDailyAgent()` 工厂函数
- `src/agents/daily/tools/open-app.ts` — 新建，`open -a "AppName"` 打开 macOS 应用
- `src/agents/daily/tools/open-url.ts` — 新建，打开 URL 或拼接 Google 搜索链接
- `src/agents/daily/tools/run-shell.ts` — 新建，执行任意 shell 命令（复用 execute-command 的超时+黑名单逻辑）
- `src/agents/daily/tools/index.ts` — 新建，导出 `ALL_DAILY_TOOLS`
- `src/cli/index.ts` — 重构为全干哥（QuanGanGe）主 Agent 入口，注册 `coding_agent` 和 `daily_agent` 两个工具
- `src/cli/display.ts` — 标题栏文案从"Coding Agent"改为"QuanGanGe · 全干哥"
- `src/cli/tools/`（目录）— 删除，已迁移至 `src/agents/coding/tools/`

**实现要点：**

子 Agent 无状态设计：每次工具调用都新建实例，不保留跨次调用的对话历史。任务上下文全部通过 `task` 参数由主 Agent 描述传递。

```typescript
// 主 Agent 注册 coding_agent 为工具
agent.registerTool(codingAgentToolDef, async (args: { task: string }) => {
  const codingAgent = createCodingAgent(client, CWD, subAgentCallbacks);
  return await codingAgent.run(args.task);  // 每次新建，无状态
});
```

TUI 透传：子 Agent 的 `onToolCall` / `onToolResult` 回调复用主界面的 `printToolCall` / `printToolResult`，子 Agent 内部工具调用在同一 TUI 中可见。主 Agent 路由子 Agent 时，额外展示路由标签（`💻 Coding Agent ← 路由到`），让用户清楚看到调度决策。

`/tools` 命令增强：同时列出 coding 和 daily 所有工具，并标注归属：
```
[coding] read_file
[coding] write_file
...
[daily]  open_app
[daily]  open_url
[daily]  run_shell
```

### 3. TTS 语音朗读可中断机制

**背景/问题：** QuanGan 的语音模块（`src/voice/`）赋予了 Agent「耳朵」（ASR 语音识别）和「嘴巴」（TTS 语音朗读）。TTS 使用 macOS 内置 `say` 命令将 Agent 回复朗读出来，在语音模式（`/voice`）下自动触发。但最初实现用 `exec` 启动 `say`，启动后进程与 Node.js 完全脱钩：关闭程序（Ctrl+C）后 `say` 仍在播放；开始录音时上一句未播完也会被录进麦克风；没有任何方式中途打断朗读。

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| `pkill say` shell 命令 | 实现简单 | 会杀掉系统中所有 `say` 进程，副作用大 |
| **`spawn` 保留子进程引用（选用）** | 精确控制单个进程，SIGTERM 定向终止 | 需要改写 exec → spawn，跟踪 currentProcess |

选型理由：`spawn` 返回 `ChildProcess` 对象，可以随时调用 `.kill('SIGTERM')`，精准停止当前朗读进程而不影响其他进程。

**涉及文件：**
- `src/voice/tts.ts` — 从 `exec` 改为 `spawn`，新增模块级 `currentProcess` 变量和 `stopSpeaking()` 导出函数；`speak()` 内部调用 `stopSpeaking()` 防止上句叠加
- `src/voice/index.ts` — 导出新增的 `stopSpeaking`
- `src/cli/index.ts` — 在三处调用 `stopSpeaking()`：`rl.on('close')`（程序退出）、`process.on('SIGINT')`（Ctrl+C）、`handleVoiceInput()` 开始录音前

**实现要点：**

```typescript
// tts.ts 核心模式
let currentProcess: ChildProcess | null = null;

export function stopSpeaking(): void {
  if (currentProcess) {
    currentProcess.kill('SIGTERM');
    currentProcess = null;
  }
}

export async function speak(text: string): Promise<void> {
  stopSpeaking();  // 先停上一句
  const proc = spawn('say', ['-v', 'Ting-Ting', escaped]);
  currentProcess = proc;
  proc.on('close', () => {
    if (currentProcess === proc) currentProcess = null;  // 避免误清
    resolve();
  });
}
```

进程引用比对（`currentProcess === proc`）是关键：防止 `stopSpeaking()` 已将 `currentProcess` 置 null 后，旧进程的 `close` 事件又把新进程的引用误清。

### 4. Daily Agent 新增 AppleScript 工具及系统提示升级

**背景/问题：** Daily Agent 负责处理非编码的日常任务（打开应用、搜索网页、执行系统命令等），是 QuanGan 成为「通用个人助手」的关键模块。用户希望不只是「打开 QQ 音乐」，还能进一步指挥 Agent 搜索并播放指定歌曲。实测中 Agent 收到「搜索并播放迪迦奥特曼」的指令后，只回复了 AppleScript 代码和操作建议，完全没有实际执行。根因有二：① Daily Agent 工具箱里没有 AppleScript 专用工具，Agent 不知道可以通过 `run_shell` + `osascript` 来控制 app；② 系统提示过于宽松，没有明确要求 Agent 主动调用工具而非给出建议。

**技术选型（AppleScript 执行方式）：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| `osascript -e "脚本"` | 无需临时文件 | 只支持单行；多行脚本报语法错 (-2741/-2740) |
| **写临时文件后 `osascript 文件路径`（选用）** | 支持任意多行脚本，行为与 Script Editor 完全一致 | 需要额外的文件 IO | 

选型理由：`-e` 标志设计仅供单行命令，多行 AppleScript 必须通过文件执行，且临时文件在 `finally` 块中清理，没有资源泄露。

**涉及文件：**
- `src/agents/daily/tools/run-applescript.ts` — 新建；脚本写入 `/tmp/quangan-as-<timestamp>.applescript`，`osascript` 执行后 `finally` 清理；辅助功能权限错误（error 1002）单独处理并给出授权引导
- `src/agents/daily/tools/index.ts` — 注册 `run_applescript` 工具
- `src/agents/daily/index.ts` — 系统提示全面重写为 `DAILY_SYSTEM_PROMPT` 常量；核心新增：三级控制策略（URL Scheme → AppleScript UI 自动化 → AppleScript 字典控制）、QQ 音乐 `qqmusicmac://` URL scheme 说明、强调"必须直接执行操作而非给建议"

**实现要点：**

```typescript
// run-applescript.ts 核心执行逻辑
const tmpFile = join(tmpdir(), `quangan-as-${Date.now()}.applescript`);
try {
  writeFileSync(tmpFile, script, 'utf-8');
  const output = execSync(`osascript ${JSON.stringify(tmpFile)}`, { timeout: 30000 });
  return output.trim() || '✅ AppleScript 执行成功（无输出）';
} finally {
  try { unlinkSync(tmpFile); } catch {}
}
```

QQ 音乐 URL scheme 通过逆向 `Info.plist` 发现：`CFBundleURLSchemes = ["qqmusicmac"]`，搜索命令为 `open "qqmusicmac://search?word=歌名（URL encoded）"`，无需辅助功能权限。

## 关键收获

1. **Sub-agents as Tools 的核心优势**：把子 Agent 当工具注册，主 Agent 的 ReAct 循环原生支持多 Agent 调度，无需额外协议；子 Agent 调用结果直接进入主 Agent 的消息上下文，链式任务自然支持。

2. **无状态子 Agent 的适用场景**：对于任务边界清晰、单次交互可完成的任务，无状态子 Agent（每次新建实例）是最简设计；主 Agent 持有长期上下文，子 Agent 专注一次性执行，职责清晰。

3. **工厂函数模式的好处**：用 `createCodingAgent()` / `createDailyAgent()` 封装 Agent 创建逻辑，调用方只传必要参数（client、workDir、callbacks），子 Agent 的内部配置细节对主 Agent 透明；未来扩展新 Agent 只需新增工厂函数，`index.ts` 改动极小。

4. **目录组织原则**：工具文件跟随 Agent 归属放在 `src/agents/<name>/tools/` 下，而非放在 `cli/` 目录；CLI 层（`src/cli/`）只负责交互，不持有业务逻辑，职责分离更清晰。

5. **进程引用跟踪模式（可中断后台任务）**：用模块级变量持有子进程引用，`stopSpeaking()` 定向 SIGTERM，是"可中断后台任务"的通用模式，适用于任何需要随时打断的 spawn 子进程（TTS、长时间 shell 命令等）。

6. **osascript 多行执行陷阱**：`osascript -e "..."` 只处理单行，多行 AppleScript 必须写入文件后用 `osascript <文件路径>` 执行。在工具层统一处理此细节，上层（LLM）只需提供脚本内容即可。

7. **系统提示决定 Agent 行为模式**：Agent"只给建议不动手"是系统提示不够明确的结果，而非模型能力问题。在提示中用"必须直接执行"、列出具体工具和调用示例，可以显著改变 Agent 的执行意愿。
