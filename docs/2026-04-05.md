### 1. 终端钢琴演奏功能

**背景/问题：** 用户希望 Agent 具备钢琴演奏能力，在终端中显示钢琴键盘 UI，当用户说"帮我弹一首歌"时，琴键随旋律高亮并播放对应音调。功能定位在 Daily Agent 中，作为日常娱乐交互能力。

**技术选型（音频播放）：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| `say` 命令（macOS 内置） | 无需额外依赖 | 只能朗读文本，无法播放精确音调 |
| **speaker npm 包（选用）** | 原生 Node.js PCM 音频播放，精确控制频率和时长 | 需要 npm install，依赖系统音频驱动 |
| `aplay` / `ffplay` 外部命令 | 功能强大 | 跨平台兼容性差，需要外部二进制 |

选型理由：`speaker` 包提供纯 Node.js 的 PCM 音频播放能力，可以精确控制正弦波频率和 ADSR 包络，适合钢琴音符播放。

**技术选型（编曲数据源）：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| LLM 实时编曲 | 支持任意歌曲，无需外部数据 | 质量不稳定，速度慢，音符可能不准确 |
| **先 web_search 搜索曲谱，解析失败再 LLM 编曲（选用）** | 优先使用现成准确曲谱，fallback 保证可用性 | 需要两次 LLM 调用（搜索+解析） |
| 本地硬编码预设曲目 | 快速、准确 | 只支持有限歌曲，无法扩展 |

选型理由：网络曲谱质量远高于 LLM 编曲，搜索优先策略确保准确性，LLM 作为 fallback 保证对冷门歌曲的支持。

**技术选型（钢琴 UI 渲染）：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| ANSI 清屏 + 重绘 | 原地更新，不刷屏 | 与 Ink 组件模型冲突，React 状态管理失效 |
| **纯 Ink Box + Text 组件（选用）** | 与现有 UI 框架一致，支持 React 状态驱动 | Ink 会在 Static 历史区追加内容，需注意布局 |
| 外部终端库（如 blessed） | 功能强大 | 引入新依赖，与 Ink 冲突 |

选型理由：Ink 是项目已有的 UI 框架，使用 Box + Text 组件保持技术栈统一，通过条件渲染（钢琴 UI 替代输入区）避免历史区刷屏问题。

### 2. Voice 模式重复触发修复

**背景/问题：** Voice 模式下按 Enter 开始录音时，同一语音被连续触发两次，导致重复识别和重复处理。用户看到"语音识别中..."出现两次，同一条语音被处理两遍。

**根因分析：** Enter 键被两个地方同时处理：
1. `useInput` 钩子监听到 `key.return` → 调用 `onSubmit('')`
2. `TextInput` 组件的 `onSubmit` 也被触发 → 调用 `handleSubmit` → 也调用 `onSubmit('')`

每次按 Enter 都会调用两次 `handleVoiceInput`。

**技术选型：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 防重入标志（`_isProcessingVoice`） | 简单可靠，任何重复触发都能拦截 | 如果第一次处理失败，标志可能卡住 |
| **去除重复事件源（选用）** | 从根源解决，不依赖状态管理 | 需要确认 TextInput 在空输入时会触发 onSubmit |
| 防抖（debounce） | 通用方案 | 延迟响应，体验差 |

选型理由：Enter 重复触发是事件源冲突问题，应该从根源去除重复监听，而不是用状态掩盖问题。保留 `TextInput` 的 `onSubmit` 处理，去掉 `useInput` 中的 Enter 监听。

**涉及文件：**
- `src/cli/ui/App.tsx` — 去掉 `useInput` 中对 `key.return` 的处理，Voice 模式 Enter 完全交给 `TextInput` 的 `onSubmit` 处理；添加调试日志验证是否还有重复调用
- `src/cli/index.ts` — 添加 `_isProcessingVoice` 全局防重入标志作为第二层保护，`handleVoiceInput` 开头检查标志，所有 return 分支和 finally 中重置标志

**实现要点：**

第一层保护：去除重复事件源
```typescript
// App.tsx - 去掉 useInput 中的 Enter 处理
useInput((_, key) => {
  if (key.escape && isRunning) {
    onAbort();
  }
  // voice 模式下 Enter 触发录音（但不在这里处理，交给 TextInput 的 onSubmit，避免重复）
});
```

第二层保护：防重入标志
```typescript
// index.ts
let _isProcessingVoice = false;

async function handleVoiceInput(): Promise<void> {
  if (_isProcessingVoice) {
    console.log('[voice] 忽略重复触发');
    return;
  }
  
  _isProcessingVoice = true;
  try {
    // ... 录音、识别、处理
    await processMessage(text);
  } finally {
    _isProcessingVoice = false;
  }
}
```

双层保护确保即使 `TextInput` 行为异常，也不会出现重复处理。

## 关键收获

1. **依赖注入解决循环引用**：当工具需要访问全局 store 或 client 但直接 import 会导致循环依赖时，使用 `initXxxTool(store, client)` 在启动时注入依赖，工具内部通过模块级变量持有引用，是 Node.js 项目解决循环引用的通用模式。

2. **ADSR 包络优化合成音色**：纯正弦波音色刺耳不自然，添加 ADSR（Attack-Decay-Sustain-Release）包络模拟真实乐器的音量变化曲线，20ms Attack 模拟琴槌击弦、30ms Decay 到 0.7 Sustain 模拟余音、50ms Release 模拟制音，让合成音色更接近真实钢琴。

3. **搜索优先 + LLM Fallback 策略**：对于有现成数据源的任务（如曲谱），优先搜索网络获取准确数据，解析失败再降级到 LLM 生成。这种策略兼顾准确性和覆盖率，比纯 LLM 生成更可靠。

4. **Ink 组件布局避免历史区刷屏**：在 Ink + Static 组件架构中，动态更新的 UI（如钢琴键盘）不应放在 Static 历史区内，而应作为条件渲染的独立区域，通过替换输入区等方式显示，避免每次状态更新都追加新行到历史区。

5. **双层防护处理重复事件**：对于用户交互事件（如按键），第一层从根源去除重复事件源（去掉多余的监听器），第二层用防重入标志兜底，双重保护确保鲁棒性。

6. **音符白名单校验**：LLM 返回的音符名称可能不标准（如 "do"、"re"、"1"），在解析后必须用白名单校验过滤，只接受标准格式（C4-C6、REST），避免无效音符导致播放错误。
